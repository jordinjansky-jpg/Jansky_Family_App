# UI Rework — Phased Plan

**Date:** 2026-04-19
**Status:** Proposed
**Spec:** [docs/DESIGN.md](../../DESIGN.md)
**Rules digest:** [CLAUDE.md](../../../CLAUDE.md) → *Design Rules*
**Mockups:** [mockups/](../../../mockups/)

---

## Purpose

Bring the live app into alignment with the new Design Spec without destabilizing production. The spec changes layout, component density, and naming across most pages; doing it in one landing would be high-risk and high-rollback-cost. This plan sequences the work into shippable phases, each of which can ship to `main` and be auto-deployed by Cloudflare without breaking existing users.

## Workflow discipline

**Plan-as-memory.** When any phase spec is written (or any phase's build reveals a scope change), update this plan file in the **same PR** so future-you reading the specs fresh in 6 months sees a coherent picture. The plan is the running look-ahead; specs are per-phase contracts; never let them drift apart.

**Cadence:** plan (done) → spec phase N → build phase N → update plan if scope deviated → spec phase N+1 (review phase N first) → build phase N+1 → …

**Pre-phase-N+1 review:** before writing spec N+1, run the review gate at the bottom of spec N (all exit criteria met, grep recipes pass, deferred register updated, baselines refreshed, this plan file current).

## Ground rules

- **Firebase data stays stable.** No schema changes in Phases 0–6. Schema changes are flagged at the phase that introduces them (currently Phase 7+, backlog-driven).
- **No big-bang refactors.** Each phase is shippable alone and leaves the app functional.
- **Tokens before components. Components before pages.** Restyling a page before the token/component pass just churns CSS.
- **Phone first, every phase.** Tablet and kiosk pick-ups happen after the phone cleanup is stable.
- **Preserve existing behavior.** No feature removal without a named call-out in the PR.
- **Bump SW cache (`sw.js` `CACHE_NAME`) on every phase** that ships. Missing this is the #1 source of "the update didn't stick" reports.
- **Test matrix per phase:** 375px (phone), 768px (tablet portrait), 1024px (tablet landscape). Light + dark theme. Real data in Jansky family Firebase.

## Risk bar for each phase

Each phase lists:
- **Goal** — what the user experience looks like after.
- **Files** — concrete files touched.
- **Migration** — data changes, if any.
- **Exit criteria** — observable things that must be true to call it done.
- **Rollback plan** — how we revert cheaply if we regret it.

## Phase 0 — Foundation (tokens, base, housekeeping)

**Goal:** Every downstream phase operates on a clean token and base layer. After this phase the app looks ~95% the same as today; the scaffolding just became spec-compliant.

**Files:**
- `styles/base.css` — add/align tokens: `--spacing-xs/sm/md/lg/xl/2xl`, `--font-xs…3xl`, `--radius-sm/md/lg/xl/full`, elevation, motion, z-index bands per §3 of DESIGN.md.
- `styles/components.css` — retire any hardcoded hex; replace with tokens. No visual redesign yet.
- `shared/components.js` — audit for inline styles baked into HTML strings; move to classes.
- Sweep all pages (`index.html`, `calendar.html`, `scoreboard.html`, `tracker.html`, `admin.html`, `kid.html`, `setup.html`, `person.html`) for `style="…"` inline — move to classes.
- Replace all `window.confirm` / `window.alert` callsites with `showConfirm()` and toasts.
- Add `prefers-reduced-motion: reduce` guard to any animation-heavy CSS that's missing it.
- `sw.js` — bump cache version.

**Migration:** None. Token renames done via CSS-var aliases (keep old names as alias to new for one phase; remove in Phase 1).

**Exit criteria:**
- Grep for `style="` returns 0 matches in files listed in the Phase 0 spec's §2.1 scope (`setup.html`, `person.html`, `shared/components.js`, `shared/calendar-views.js`, `dashboard.js`). HTML files slated for later-phase rework keep their inline styles until that phase; see the Phase 0 spec's §2.4 deferred tech-debt register for the full deferral map.
- Grep for `window.confirm` / `window.alert` returns 0.
- Grep for `#[0-9a-f]{3,6}` inside `styles/components.css` returns 0 matches that aren't tokens.
- App still looks and behaves identically in manual smoke: dashboard load, complete a task, open calendar day, open kid mode, admin PIN.
- Lighthouse accessibility score not lower than before.

**Rollback:** Revert the PR. No data changes.

## Phase 1 — Dashboard rework

**Goal:** Dashboard matches [mockups/01-dashboard.html](../../../mockups/01-dashboard.html). Calm header (title + subtitle), banner slot, optional ambient strip, Events → Today cards with `.card--done` sink, FAB, 5-tab bottom nav.

**Files:**
- `index.html` — restructure markup to match mockup. Header becomes `.app-header` with subtitle.
- `dashboard.js` — adapt rendering to the new card structure (slot-based `.card` with `.card__leading/body/trailing`). Keep data flow identical. Wire the optional Ambient Strip behind a settings toggle (default off on phone; see Phase 3 Settings for toggle UI).
- `shared/components.js` — add/align Card variants (`--task`, `--event`, `--meal`, `--done`) per §5.1 of DESIGN.md.
- `styles/dashboard.css` — align to tokens; drop bespoke spacing.
- `styles/layout.css` — header + bottom-nav refinements. **The 5-tab bottom nav (Home · Calendar · Scores · Tracker · More) and its More menu must include a "Rewards" entry that opens today's existing Store sheet — no behavior change, just correct routing so Phase 6 can swap the target without moving the entry point.**
- `sw.js` — bump cache.

**Migration:** None. Ambient strip toggle reads from existing `rundown/settings` — if the field isn't present, default off.

**Exit criteria:**
- Dashboard visually matches the mockup on 375px.
- Single banner only at any one time (queue logic in `dashboard.js`).
- Completed tasks sink to bottom (`.card--done`).
- All interactive elements ≥ 44×44.
- Bottom nav shows 5 tabs exactly per DESIGN.md §2. **More menu lists a "Rewards" row that opens the current Store sheet** (destination-stable for Phase 6; no routing rework expected).
- No regressions: overdue banner still fires, celebration still plays, real-time listener still updates.
- Inline styles in the phase's primary HTML file cleared — this page's row in the Phase 0 spec §2.4 deferred register is satisfied. (Applies to `index.html`.)
- Raw hex in the phase's primary page CSS file replaced with tokens — this file's row in the Phase 0 spec §2.4 deferred register is satisfied. (Applies to `styles/dashboard.css`.)

**Rollback:** Revert the PR. Schema is untouched.

## Phase 6 — Rewards Store unification (adult/kid parity)

**Goal:** The Rewards Store becomes a first-class destination with three intent-specific entry points, all opening the same component. Adult and kid experiences use identical layout, tabs, and cards — only the post-Redeem flow differs (immediate vs approval-gated). Matches DESIGN.md §6.7 after the 2026-04-23 revision.

**Problem this fixes:** Today the Store is reachable only through Scoreboard on the adult side, dressed as a kid-mode feature bolted onto the leaderboard. Adults don't live on Scoreboard — it's a "how am I doing" surface, not transactional. Whole subsystems (Bank, Wishlist, approvals, redemption history) are buried behind a low-traffic tab. Adults and kids see visibly different Store shells even though the DB model is identical.

**Files:**
- `scoreboard.html` + `dashboard.js` — no new Store code; Phase 1 already routes "Rewards" in the More tab to today's Store sheet. This phase swaps the destination to the new unified page/sheet.
- New file OR existing: `rewards.html` **or** Store moves fully into `shared/components.js` as a sheet callable from any page. Decision at start of phase. Preference: sheet-based so adults and kids share the same open path from More/Scoreboard CTA/Bell deep-link.
- `shared/components.js` — add `.card--reward` variant per §5.1 if not already present; ensure Tab component supports the Custom | Functional | Bounties | Wishlist | Bank ordering.
- `shared/firebase.js` — no schema changes. Audit reward/bank/wishlist helpers for any adult-vs-kid branching that can be collapsed.
- `styles/components.css` — reward card variant tokens.
- `styles/scoreboard.css` — demote the current "Store" section to a balance card + "Open Store" CTA row (contextual entry); remove the in-place Store rendering.
- Bell dropdown (in `shared/components.js` `initBell()`) — tighten deep-links: approval-request notifications open Store → Approvals view; redemption outcomes open Store → relevant tab; bank arrivals open Store → Bank.
- Kid mode Store tile — confirm it opens the same unified component, not a parallel kid-only path.
- Admin Rewards — unchanged in this phase (admin authoring flow is separate from the user-facing Store).
- `sw.js` — bump cache.

**Migration:** None. `rewards/`, `bank/`, `wishlist/`, `messages/`, `balanceAnchors/` nodes untouched. This is a UI unification, not a schema change.

**Exit criteria:**
- Rewards Store opens the **same component** from all four routes: More tab (phone) / left rail (tablet after Phase 7), Scoreboard balance CTA, Bell deep-link, Kid Store tile.
- Scoreboard no longer renders Store contents inline — only a balance card with a CTA.
- Adults see the same tabs, card pattern, and balance display as kids. Flow differences (approval gate) are behavioral only, not layout-level.
- Bell deep-links land on the correct Store tab for each message type.
- Custom reward Redeem (kid) still routes to parent approval; functional reward Redeem still immediate; Bank use still adult-immediate / kid-approval-gated.
- Redemption history, Wishlist, Bounties all reachable by both audiences from the same Store entry.
- No regressions in reward flows (bonus/deduction messages, multipliers, redemption counter, achievement unlock on redeem).
- Inline styles cleared in touched HTML; raw hex replaced with tokens in touched CSS (satisfies remaining Phase 0 §2.4 rows for these files if any).

**Rollback:** Revert the PR. `rewards/`, `bank/`, `wishlist/` data preserved (no writes changed). Scoreboard re-renders the Store sheet from its prior location.

**Dependencies:** Phase 1 (More-tab Rewards entry stub must exist). Phase 5 (Scoreboard standardized so the balance card fits cleanly). Ideally before Phase 7 (Tablet) so the two-pane Store layout is designed once, not retrofitted.

**Open questions (resolve at phase start):**
- Standalone `rewards.html` page vs sheet-only from any page? Preference: sheet + dedicated route, so the More-tab link and Bell deep-links both produce a shareable URL, while the Scoreboard CTA can open as a sheet without leaving the Scores context.
- Approvals view UX: inline Approvals tab in Store vs continuing to live in the Bell dropdown? Likely both — Bell for quick triage, Store Approvals tab for bulk/history.

## Ordering rationale

The phase order is deliberate:

- **Phase 0 before anything** because restyling on top of hardcoded hex and inline styles doubles the work.
- **Dashboard (1) before Calendar (2)** because more people open the app to the dashboard than the calendar; a polished dashboard buys us goodwill during the longer calendar rebuild.
- **Admin (3) after Dashboard/Calendar** because the People button-soup is painful but nobody lives in admin daily — it's the highest-impact-per-hour-of-work inside admin, but lower urgency than the two daily screens.
- **Kid mode (4) after shared component catalog is solid** because kid mode's "parallel CSS ecosystem" issue is fixable only if the modifier pattern is already proven on core pages.
- **Scoreboard/Tracker (5) last among phone surfaces before rewards** because they're already the closest to compliant — minor tab cleanup vs. the bigger Dashboard/Calendar/Admin redos — and Phase 6 needs a clean Scoreboard to host the balance card + Store CTA.
- **Rewards Store unification (6) after Scoreboard** so the balance card, CTA placement, and Bell deep-links land into a Scoreboard and Bell that are already spec-aligned. Running it earlier would mean designing the Store's Scoreboard integration against a shell we're about to rework.
- **Tablet (7) after all phone work is stable** so we don't move two targets at once. This way, Phase 7 designs the tablet two-pane for a Store that already has its final shape, instead of retrofitting the unification inside a tablet layout.
- **Kiosk (8) last** because it's a new file with no user impact if it's delayed; it also benefits from having Weather + Meals shipping on the current screens first.

## Ship cadence

- Each phase targets **one PR, one deploy**. Cloudflare auto-deploys on push to `main`. No feature flags in code — the spec is the flag; we gate by "does the next phase's spec section still allow this".
- **SW cache bumps are non-negotiable.** Add a `CACHE_BUMPS.md` comment in `sw.js` for each phase so future readers can correlate cache versions to phases.
- If a phase reveals a spec gap, update `docs/DESIGN.md` in the same PR. Never let drift compound.

## When to re-plan

Rewrite this document (bump the date in the filename) when:
- A backlog item changes priority and reorders Phase 8+.
- A phase takes >2x the estimated sessions — that's signal the phase was too large; split it.
- The spec changes a non-negotiable (a "Do not" rule moves). Phases that touched that rule may need follow-up.

## Open questions (resolve at phase start)

- **Phase 3:** dedicated `admin-person.html` route vs. in-`admin.html` view-switcher? Current preference: in-`admin.html` for session/PIN reuse. Revisit when scoping the phase.
- **Phase 6:** standalone `rewards.html` route + sheet, sheet-only, or both? Preference: both — direct URL for More-tab and Bell deep-links, sheet open path for Scoreboard CTA. Decide at phase start.
- **Phase 6:** do adult redemptions still log an audit entry (bonus/deduction message to self) for family visibility, or only kid redemptions? Decide at phase start.
- **Phase 7:** is the left rail collapsible on tablet portrait (768–1023px) or does it stay fixed? Mockup not yet drawn. Decide at phase start.
- **Phase 8:** kiosk pairing — does the display bind to a family by URL parameter, QR setup flow, or Firebase auth? Decide when scoping; likely URL parameter + hidden admin pairing.
